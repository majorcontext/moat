# AI Quickstart and Agent Context Injection

## Overview

Two features that make Moat smarter about AI agents:

1. **`moat <agent> quickstart`** — Auto-generate a `moat.yaml` for an existing project by running the agent itself in a bootstrap container to analyze the codebase.
2. **Agent Context Injection** — Give agents running inside Moat awareness of their environment (grants, services, network policy, MCP servers) via native instruction files and environment variables.

## Feature 1: `moat <agent> quickstart`

### Command

```
moat claude quickstart
moat codex quickstart
moat gemini quickstart
```

### Flow

1. User runs `moat claude quickstart` in their project directory.
2. Moat builds a minimal bootstrap container:
   - Workspace mounted read-only at `/workspace`
   - Agent CLI installed (claude-code, codex-cli, or gemini-cli)
   - Agent credential required (anthropic/openai/gemini grant)
   - No services, no other grants, no hooks
3. Agent runs with a structured prompt containing auto-generated schema and dependency references.
4. Agent analyzes manifest files (`package.json`, `go.mod`, `pyproject.toml`, `Gemfile`, `docker-compose.yml`, `.env.example`, README, source code).
5. Agent outputs a `moat.yaml` to the workspace.
6. Container exits. User reviews and runs `moat run`.

### Prompt Content

The agent receives three embedded artifacts:

| Artifact | Source | Generated by |
|----------|--------|-------------|
| Schema reference | `config.Config` Go struct | `go generate` reflection |
| Dependency reference | `deps/registry.yaml` | `go generate` YAML parse |
| Prompt template | Hand-written | Embeds the above two |

#### Schema Reference (auto-generated at build time)

Generated from `internal/config/config.go` struct tags and types. Produces a condensed markdown reference of all moat.yaml fields, their types, defaults, and descriptions.

```go
//go:embed schema_reference.md
var schemaReference string
```

#### Dependency Reference (auto-generated at build time)

Generated from `internal/deps/registry.yaml`. Lists all available dependencies with their types, versions, and descriptions. Equivalent to `moat deps list` output.

```go
//go:embed deps_reference.md
var depsReference string
```

#### Prompt Template

The prompt combines schema, deps, and instructions:

- Role: "You are a Moat configuration expert."
- Schema reference (embedded)
- Dependency reference (embedded)
- Dynamic prefix documentation: `npm:pkg`, `pip:pkg`, `uv:pkg`, `cargo:pkg`, `go:pkg`
- Escape hatches: `hooks.post_build_root` for system packages not in the registry (e.g. `apt-get install -y direnv`)
- 2-3 condensed examples from `examples/`
- Task: "Analyze the project at /workspace. Look at manifest files, source code, READMEs, docker-compose files, .env files. Generate a minimal moat.yaml."
- Output format: "Output only the YAML, no explanation."

### Implementation Location

```
internal/quickstart/
  quickstart.go         — Bootstrap container creation, prompt assembly, output capture
  generate.go           — go:generate script for schema + deps references
  schema_reference.md   — Generated (do not edit)
  deps_reference.md     — Generated (do not edit)
  prompt.md             — Hand-written prompt template

cmd/moat/cli/quickstart.go  — cobra subcommand under each agent command
```

### Bootstrap Container Config

The quickstart uses a minimal hardcoded config (no moat.yaml needed):

```go
cfg := &config.Config{
    Name:         "quickstart",
    Dependencies: []string{"claude-code"}, // or codex-cli, gemini-cli
    Grants:       []string{"anthropic"},    // or openai, gemini
}
```

The agent runs with `-p` (prompt mode) to execute the analysis prompt non-interactively.

## Feature 2: Agent Context Injection

### Architecture

```
internal/context/
  context.go    — RuntimeContext struct, populated from run config + resolved state
  render.go     — Renders RuntimeContext to markdown string
```

Each agent provider calls `context.Render()` and writes the result to the agent's native instruction file path in its staging directory.

### RuntimeContext Struct

```go
type RuntimeContext struct {
    RunID         string
    Agent         string
    Workspace     string
    Grants        []Grant          // name + description of what it provides
    Services      []Service        // name, version, env vars, connection URL
    Ports         []Port           // name, container port, host port env var
    NetworkPolicy NetworkPolicy    // policy type + allowed hosts
    MCPServers    []MCPServer      // name + description
    Mounts        []Mount          // target path, read-only flag
    Volumes       []Volume         // name, target path
}
```

### Rendered Markdown Output

```markdown
# Moat Environment

You are running inside a Moat sandbox — an isolated container with
credential injection and network controls.

## Workspace
- Path: /workspace
- Mount: read-write

## Grants
- `github` — GitHub access via `gh` CLI. Credentials are auto-injected
  at the network layer; you will never see raw tokens.
- `anthropic` — Anthropic API access via proxy.

## Services
- PostgreSQL 17 available at `$MOAT_POSTGRES_URL`
- Redis 7 available at `$MOAT_REDIS_URL`

## Network Policy
- Policy: strict
- Allowed hosts: api.github.com, *.npmjs.org

## MCP Servers
- `github` — GitHub tools (issues, PRs, search)

## Ports
- `api` (8080) — host port at `$MOAT_API_HOST_PORT`

## Run Metadata
- Run ID: abc123
- Agent: claude
```

Sections are only included when relevant (no empty "Services" section if there are no services).

### Delivery Per Agent

| Agent | Instruction file | Staging path | Copied by |
|-------|-----------------|-------------|-----------|
| Claude | `~/.claude/CLAUDE.md` | `/moat/claude-init/CLAUDE.md` | moat-init.sh |
| Codex | `~/.codex/AGENTS.md` | `/moat/codex-init/AGENTS.md` | moat-init.sh |
| Gemini | `~/.gemini/GEMINI.md` | `/moat/gemini-init/GEMINI.md` | moat-init.sh |

Each provider's `PopulateStagingDir()` method generates the context markdown and writes it alongside existing config files. The existing init script copies everything to the home directory.

### Environment Variables

Always set regardless of agent type:

```
MOAT_RUN_ID=abc123
MOAT_AGENT=claude
MOAT_GRANTS=github,anthropic
MOAT_SERVICES=postgres,redis
MOAT_NETWORK_POLICY=strict
MOAT_WORKSPACE=/workspace
```

These are for non-LLM tools or scripts that want to detect they're running inside Moat.

### Implementation Changes

1. **New package:** `internal/context/` with `RuntimeContext` struct and `Render()` function.
2. **Provider updates:** Each provider's `PopulateStagingDir()` calls `context.Render()` and writes the markdown file.
3. **Init script update:** Add the instruction file to the copy list in `moat-init.sh`.
4. **Environment variables:** Added in `internal/run/manager.go` alongside existing env var injection.

## Scope Exclusions

- No `apt:` dynamic prefix (separate effort)
- No dynamic context updates during a run (static at container start)
- No custom context templates (one format for all agents)
- No agent-writable context (read-only)
- No new-project scaffolding (`quickstart` is for existing projects only)
